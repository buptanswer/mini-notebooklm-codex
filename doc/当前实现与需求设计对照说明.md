## 1. 当前已完成的主链路

当前仓库已经完成 stage1 到 stage5 的基础主链路，并且与项目当前文档保持一致：

1. `React + FastAPI` 前后端分离工程
2. `SQLite + Qdrant + 文件系统` 数据底座
3. MinerU SaaS 文档/图片解析与 `document_ir.json`
4. 原始输出严格审计、`warning` 保留、`needs_review` 标记
5. `document_ir_enriched.json` 多模态富化
6. DOM 重建、Parent/Child chunking
7. `text-embedding-v4` 向量化与 Qdrant 入库
8. SQLite FTS5 关键词检索
9. `qwen3-rerank` 重排序
10. `qwen3.5-plus` 普通问答与 SSE 流式问答
11. 前端上传、任务状态、聊天问答、引用资料面板、PDF 高亮溯源
12. 知识库重命名/删除、文件与文件夹重命名/移动/删除、批量删除、目录树浏览

## 2. 目前保留的边界

当前仍然保留的边界只有一类：

1. 音视频链路暂缓
   - stage6 作为提升项延期
   - 当前不支持飞书妙记、通义听悟、音频转写文本入库、视频关键帧
   - 这一点已经在需求文档中明确说明，不再视为“实现遗漏”

## 3. 实现中与早期草案不同但已文档化的取舍

以下内容属于“实现阶段的工程化调整”，不是缺失功能：

1. 前端技术栈采用 `React + Vite + TypeScript + 自定义 CSS`
   - 没有引入 `Shadcn UI + Tailwind CSS`
   - 原因是当前项目页面规模可控，优先保证上传、任务、问答、溯源和文件管理的完整闭环
   - 这样可以减少依赖和样式体系迁移成本，避免在结题阶段因为 UI 框架切换影响主链路稳定性
2. Raw MinerU 层默认采用“warning 并保留”的策略
   - 未知字段、未知 block、fallback shape 会写入 warning
   - 默认不会因为 warning 直接中断整篇文档
   - 用户侧通过 `needs_review` 黄标感知风险，再决定是否人工复核
3. 输出目录结构没有照搬最初示意图
   - 当前产物放在知识库目录树和 MinerU bundle 目录中
   - 没有单独复制出一套 `rag_pipeline_output/assets/`
   - 这样可以减少重复拷贝，降低路径同步和缓存清理成本
4. 代码模块命名没有完全按草案示例命名
   - 但职责边界仍然对应 Raw Schema / IR / Enrichment / Chunk / Adapter / Writer / Validator 等层次
   - 当前命名更贴近实际仓库结构和 FastAPI 服务拆分方式

## 4. 当前版本结论

当前仓库已经具备“从零上传资料到最终问答和溯源”的完整基础能力，适合作为课程设计基础版本结题。

同时也需要明确：

1. 音视频仍是后续提升项
2. 如需后续团队协作型 UI 规范化，可以再评估引入 `Shadcn UI + Tailwind CSS`
3. 当前版本已经优先把影响检索和问答效果的主链路补齐，包括富化、流式问答和完整文件管理
