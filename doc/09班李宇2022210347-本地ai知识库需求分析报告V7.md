# 题目: 面向大学生的多模态课程知识库与AI辅导系统（Mini-NotebookLM）

> 说明：本文是一份面向课程设计落地的需求分析文档。项目的目标不是对标或超越 Google NotebookLM、腾讯 ima 等成熟产品，而是完成一个适合本科课程设计规模的 Mini 版多模态 AI 知识库系统。
>
> 项目的重点在于：
>
> - 亲手完成一条较完整的 AI 知识库工程链路
> - 理解文档解析、结构化、切片、向量检索、重排序、多模态问答等关键环节
> - 在有限时间内做出一个效果可靠、可演示、可复盘的项目
> - 为之后的实习面试、项目介绍和技术问答积累真实经验
>
> 本文档的技术方案以当前项目内两份技术文档为依据：
>
> - [在线API输出文件格式（SaaS推断版）.md](.\在线API输出文件格式（SaaS推断版）.md)

## 1、任务概述

### 1.1、目标

本项目旨在开发一款适合大学课程学习场景的“本地化多模态个人知识库与 AI 辅导系统”。它可以理解为一个 Mini-NotebookLM：保留知识库问答系统最核心的工程主线，但控制范围和复杂度，使其适合作为程序设计实训课程中的课程设计项目。

系统采用现代前后端分离架构，在本地 Windows 环境中运行，但核心解析、向量化、重排序和问答能力依托云端 API。

本项目的目标不是追求“全面超越现有成熟产品”，而是在本科生可完成的工程范围内，围绕“基础效果不能太差、系统链路尽量完整、关键技术点能够讲清楚”构建一条高质量知识链路：

1. 先用专业解析服务把复杂资料尽可能高保真地结构化
2. 再由自研系统重建文档结构、标准化中间格式并进行高质量切片
3. 使用效果更稳定的纯文本向量模型做召回
4. 增加重排序环节提升相关性
5. 在最终回答阶段引入多模态旗舰模型，对图片、表格、公式等复杂内容进行综合理解

本项目当前的技术路线是“文本检索为主、多模态透传增强”的实用方案，而不是“所有模态统一嵌入到同一向量空间”的实验性方案。

### 1.2、系统（或用户）的特点

#### （1）目标用户与具体应用场景

预期核心用户为在校大学生，典型场景如下：

1. 课后录音与课件联合复盘
   - 学生上传课堂 PPT、老师发的 Word/PDF 材料、拍摄的板书照片
   - 后续可追加音频转写结果
   - 系统将资料统一组织成课程知识库，辅助学生做复盘、答疑与考前整理
2. 期末 PPT 冲刺与重点提问
   - 学生上传课程全套 PPT、实验讲义与辅导资料
   - 系统能够保留标题结构、图片、表格、公式及其上下文关系
   - 学生可围绕某一章节、某一张图、某个表格提问
3. 教学通知与课程杂项信息归档
   - 学生上传教学大纲、课程通知、截图照片、选课说明等碎片资料
   - 系统帮助提取考核方式、作业要求、时间节点、联系方式等信息

#### （2）产品优势与特点

本项目强调以下特点：

1. 解析优先，而非盲目向量化
   - 文档质量首先由 MinerU 提供的结构化输出保证
   - 不是把原始文档粗暴切片后直接向量化
2. 自研 IR 中间格式
   - 不直接依赖 MinerU 原始 JSON 作为知识库输入
   - 而是通过自研的 IR 标准化层重建文档树、标题路径、坐标锚点、多模态资产关系
3. 文本检索主导，多模态问答增强
   - 检索阶段使用纯文本向量 + 关键词检索 + 重排序
   - 回答阶段再使用多模态模型读取图片、表格截图等原始资产
4. 高保真引用与可视化溯源
   - 支持按 `origin.pdf` 页坐标进行高亮
   - 支持展示命中的页码、块坐标、原图资产、时间戳等信息

这些特点的目标不是与商业产品竞争，而是让项目具备：

- 明确的技术亮点
- 真实的工程复杂度
- 可用于课程展示与简历介绍的完整故事线

### 1.3、假定和约束

1. 开发约束
   - 单人结合 AI 编程助手完成开发
   - 坚持 KISS 原则，不引入 LangChain 等重封装框架
   - 优先使用原生 SDK 与透明的数据流
2. 运行环境约束
   - 项目作为本地 Web 应用交付，通过浏览器访问
   - 不做桌面打包，将精力集中在解析、检索、排序和问答效果
3. 时间约束
   - 课堂有效开发时间约 36 小时
   - 第一优先级是 MinerU 支持的基础文件类型：PDF、PPT、Word、图片
   - 音视频链路作为提升项，在基础链路完成后再接入；当前结题版本先不纳入主线实现
4. 项目定位约束
   - 本项目是课程设计，不追求覆盖所有边界场景
   - 更关注核心链路是否跑通、关键体验是否可靠、技术路线是否自洽
   - 目标是做出一个“能演示、能解释、能回答项目问题”的 Mini 版系统

## 2、需求规定

### 2.1、软件功能说明

本系统划分为六大核心模块。

#### 1. 多格式资料接入与异步解析模块

该模块负责把用户上传的原始资料转成后续知识库可处理的中间结果。

支持范围：

1. 文档与图片类
   - `pdf`
   - `ppt/pptx`
   - `doc/docx`
   - `png/jpg/jpeg`
   - 统一通过 MinerU 在线 API 解析
2. 音视频类
   - `mp3/wav/m4a/mp4` 等
   - 通过飞书妙记或通义听悟转写为带说话人与时间戳的文本文件
   - 视频关键帧抽取作为提升项，后期接入

边界说明：

- 当前项目内两份技术文档主要覆盖 MinerU 对文档与图片的解析链路
- 音视频链路在本需求文档中已经确定方向，但当前结题版本暂不实现；详细的中间格式、切片规则与入库规范后续需要单独补充

处理要求：

1. 对文档与图片资料
   - 后端上传文件到 MinerU 在线 API
   - 获取 `full_zip_url`
   - 下载并解析返回压缩包
   - 识别 `content_list_v2.json`、`layout.json`、`*_origin.pdf`、`images/*`
2. 对音视频资料
   - 获取说话人分段、时间戳与正文文本
   - 输出为知识库可消费的标准文本格式
   - 该链路在当前版本中属于提升项，基础版本先聚焦 MinerU 支持的文件类型

#### 2. MinerU 返回数据解析与标准化 IR 模块

这是新版系统的核心中间层。

系统不能直接把 MinerU 输出喂给向量库，而必须经过自研标准化处理：

1. 以 `content_list_v2.json` 为主输入
2. 以 `layout.json` 提供页面尺寸与绝对坐标补充
3. 重建文档 DOM 层级树
4. 为每个正文块注入 `header_path`
5. 提取 `page_header/page_footer/page_number/page_footnote`
6. 将图片、表格、公式、代码、列表等多模态对象统一建模
7. 结合 `*_origin.pdf` 保存可供前端高亮的页坐标锚点

标准化输出至少包括：

1. `document_ir.json`
   - 基础标准化中间格式
   - 记录文档页信息、section 树、块信息、标题路径、坐标锚点、多模态资产等
2. `document_ir_enriched.json`
   - 在基础 IR 上增加图片描述、表格摘要、上下文补充等富化结果
3. `parent_chunks.jsonl`
   - 按小节组织的大粒度块
   - 用于回答阶段回填更完整上下文
4. `child_chunks.jsonl`
   - 面向向量检索的小粒度块
   - 用于召回、重排序与精确定位

该模块必须遵循两份技术文档定义的规范：

1. 原始 MinerU 输出层允许有限兼容
2. 自研 IR 与 Chunk 层必须严格校验
3. 若结构不符合要求，应明确报错，而不是把脏数据写入知识库

#### 3. 结构感知切片与多模态富化模块

该模块负责将标准化 IR 转成适合检索的块。

关键要求：

1. 切片不能跨越 H1/H2 等逻辑边界
2. `list` 与 `code` 默认作为原子块整体保留
3. 文本段落采用 Parent/Child 双层切片
   - Child 用于向量检索
   - Parent 用于回答阶段补全上下文
4. 图片、表格、公式采用“文本代理 + 原始资产保留”的双轨策略

多模态富化策略：

1. 图片
   - 提取标题、周边段落与资产路径
   - 调用阿里 `qwen3.5-flash` 生成图像描述
   - 检索时对文本代理进行向量化
   - 回答时仍保留图片透传给最终大模型
2. 表格
   - MinerU 在较多场景下已经会返回表格 `html`、`table_caption`、`table_footnote` 和表格截图，因此系统不需要从零识别表格结构，而是在 MinerU 已有解析结果基础上继续处理
   - 优先对 MinerU 返回的 `html` 做轻量结构清洗，例如提取表头、首列、行列数、关键单元格文本
   - 再调用阿里 `qwen3.5-flash` 生成表格摘要，输出适合检索的自然语言文本
   - 若只有截图，直接调用 `qwen3.5-flash` 做视觉理解并生成摘要
   - 检索时使用摘要文本，而不是直接把整段 `html` 送去向量化
   - 回答时优先透传表格截图

说明：

- 表格摘要仍然建议通过小模型 API 完成
- MinerU 负责提供“可利用的结构化表格基础结果”，例如 `html`、caption、footnote、截图
- 我们的系统负责把这些结果进一步转成“检索友好文本”和“问答友好上下文”
- 原因是复杂表格的 `html` 直接向量化容易冗长、噪声大，而且会浪费 Token
- 先做结构清洗，再让小模型总结为“检索友好文本”，效果通常更稳定
3. 行间公式
   - 优先使用 LaTeX 文本
   - 同时保留公式截图路径用于调试或展示

#### 4. 索引、入库与混合检索模块

这一部分是与旧版差异最大的模块。

当前方案明确放弃多模态 Embedding 主检索架构，改为：

1. 纯文本向量化
   - 使用阿里云百炼纯文本向量模型 `text-embedding-v4`
   - 只对文本代理后的内容做向量化
2. 关键词检索
   - 对正文、标题、图片描述、表格摘要建立关键词索引
   - 用于补足稀有术语、专有名词与短查询场景
3. 混合召回
   - 向量召回 + 关键词召回并行进行
   - 使用融合策略合并候选结果
4. 重排序
   - 对召回结果使用阿里 `qwen3-rerank` 进行二次打分
   - 提升与用户问题的最终相关性

推荐的本地数据底座：

1. 向量库：`Qdrant`
   - 用于存储 Child Chunk 向量
   - 支持较清晰的点结构与 metadata 管理
2. 元数据与任务状态：`SQLite`
   - 存储文档信息、任务状态、文件路径、Parent/Child 关系、资产索引
   - 可结合 `FTS5` 实现本地全文检索/BM25
3. 文件系统
   - 存储原始上传文件、MinerU zip、IR JSON、Chunk JSONL、图片与 PDF 资源

说明：

- 旧版“只用单一 ChromaDB”虽然简单，但不利于结构化元数据管理、混合检索与后续工程扩展
- 新方案改为“向量库 + SQLite + 文件系统”的组合，更适合长期维护和效果优化

具体选型理由：

1. 为什么向量库优先选 `Qdrant` 而不是 `ChromaDB`
   - `Qdrant` 的点、集合和过滤条件概念更清晰，更适合后续做较复杂的 metadata 检索
   - 对 chunk、文档、知识库空间等多层级数据组织更自然
   - 后续如果要扩展混合检索、权重过滤、按知识库隔离等能力，`Qdrant` 更稳妥
   - `ChromaDB` 虽然上手简单，但在数据组织、维护性和后期扩展上不如 `Qdrant` 清晰
2. 为什么元数据与业务状态优先选 `SQLite`
   - 本项目是本地运行的课程设计，`SQLite` 部署成本低
   - 很适合存储文件树、任务状态、Parent/Child 映射、资产索引等强结构化信息
   - 可以直接配合 `FTS5` 做本地全文索引/BM25，补足关键词检索
   - 相比引入 MySQL/PostgreSQL，`SQLite` 更符合单机应用与 36 小时开发约束

#### 5. 多模态问答与提示词编排模块

该模块负责把检索结果组织为最终可用的问答上下文。

流程如下：

1. 用户在前端发起问题
2. 系统对问题做标准化与必要的查询改写
3. 执行混合召回
4. 使用 `qwen3-rerank` 重排序
5. 将命中的 Child 映射回 Parent
6. 整合标题路径、正文、表格摘要、图片描述、页码与原始图片路径
7. 调用阿里旗舰多模态模型 `qwen3.5-plus` 进行最终回答

最终回答要求：

1. 支持图文混合理解
2. 支持引用原始文档页码、段落和图片
3. 支持结合表格截图进行回答
4. 对命中内容进行高亮溯源，而非只返回纯文本摘要

#### 6. 前端交互与溯源展示模块

前端不仅要提供聊天界面，还要承担知识库文件管理、资料浏览、检索结果溯源与问答交互能力。

核心功能：

1. 支持单文件上传
2. 支持文件夹上传
3. 支持按知识库空间管理资料
4. 支持以类似网盘的方式浏览和管理知识库文件系统
5. 支持查看解析状态、切片状态、入库状态
6. 支持对话式问答
7. 支持检索命中文档的页码、块坐标、时间戳展示
8. 支持基于 `origin.pdf` 的高亮定位
9. 支持图片、表格原图、关键帧等多模态引用展示

### 2.2、对功能的一般性规定

1. 异步任务与重试
   - 文档解析、向量化、富化、音视频转写均视为异步任务
   - 对外部 API 调用可做有限次重试
   - 重试失败后应给出明确错误信息
2. 可观测性
   - 前端应能查看任务状态：排队中、解析中、切片中、向量化中、完成、失败
   - 后端应保留错误日志与关键步骤日志
3. 失败策略
   - 若原始数据不符合预期，可尝试兼容降级
   - 若 MinerU 原始输出出现未知字段，系统至少应产生 warning，并保留原始字段内容用于排查
   - 若进入 IR/Chunk 层仍不合法，必须报错并阻止入库
4. 文档引用一致性
   - 任何进入检索和回答链路的块，必须可以回溯到原文件、页码和坐标或时间戳

### 2.3、用户界面

前端采用 `React + Vite + TypeScript` 构建，当前实现使用自定义 CSS 组织工作台界面，而不是引入 `Shadcn UI + Tailwind CSS`。重点不在特定 UI 库，而在于前端不能只停留在单一聊天界面，而应设计成一个“知识库工作台”。

建议采用多页面或多工作区布局，而不是只有一个一级界面。

推荐的前端信息架构如下：

1. 知识库首页
   - 展示已有知识库空间
   - 支持新建、重命名、删除知识库
   - 展示每个知识库的文件数、更新时间、解析状态摘要
2. 知识库文件管理页
   - 采用类似网盘的界面
   - 支持单文件上传
   - 支持文件夹上传
   - 支持目录树浏览
   - 支持文件夹展开/收起
   - 支持文件删除
   - 支持文件夹删除
   - 支持文件重命名
   - 支持文件夹重命名
   - 支持文件移动或归类到其他目录
   - 支持批量选择与批量删除
   - 支持查看每个文件的解析状态、切片状态、入库状态
   - 支持按文件类型筛选
3. 对话问答页
   - 提供对话式问答界面
   - 支持 Markdown、公式、图片引用渲染
   - 支持 SSE 流式输出
   - 支持查看本次回答引用了哪些知识来源
4. 来源详情页 / 预览页
   - 点击知识库来源后进入详情页面
   - 可查看文档基本信息、标题结构、命中片段、页码、图片资产
   - 可预览 `origin.pdf`
   - 可根据 `bbox_page` 做高亮定位
5. 任务中心 / 状态页
   - 查看解析、切片、向量化、入库等后台任务
   - 显示进行中、成功、失败及错误原因

布局建议：

1. 左侧主导航栏
   - 支持收起/展开
   - 提供知识库列表、任务中心、设置入口
2. 二级侧边栏
   - 在进入某个知识库后展示该知识库下的目录树、筛选器、文件列表
   - 支持收起/展开
3. 主工作区
   - 根据当前页面显示文件管理、聊天、预览或任务内容
4. 右侧详情面板
   - 可选展示当前命中来源、页码、缩略图、表格预览、块级信息
   - 支持按需展开/收起

这样设计的好处：

1. 更符合“知识库”而不是“单聊天页”的产品定位
2. 用户可以真正管理自己的文件、目录结构和知识来源
3. 更适合展示项目的工程完整度
4. 更利于后续扩展音视频、任务队列和文档预览功能

### 2.4、对性能的一般性规定

1. 质量优先于极限低延迟
   - 系统目标是保证课程设计场景下的问答准确率与可解释性
   - 因存在混合召回、重排序和多模态推理，允许首响应略慢于极简系统
2. 参考目标
   - 一般问题的首字返回时间控制在 `3-6 秒`
   - 复杂多图、多表格问题可接受更长响应时间
3. 检索质量要求
   - 对标题、段落、图片说明、表格摘要、公式上下文具备较强召回能力
   - 相比旧版，应重点提升“长文档结构定位能力”和“复杂图表场景下的相关性”

### 2.5、数据管理能力要求

本项目采用“文件系统 + SQLite + Qdrant”的分层存储结构。

1. 文件系统
   - 原始上传文件
   - MinerU 返回 zip
   - `origin.pdf`
   - `images/*`
   - IR 与 Chunk 文件
2. SQLite
   - 文档记录
   - 文件路径
   - 任务状态
   - Parent/Child 映射
   - 资产索引
   - 可扩展使用 `FTS5`
3. Qdrant
   - Child Chunk 向量
   - 向量 metadata

要求：

1. 任何向量记录必须能映射回对应 Child Chunk
2. 任何 Child Chunk 必须能映射回 Parent Chunk 与原始文档页码
3. 任何图片/表格/公式资产必须可回溯到物理路径

### 2.6、关于合规开源组件使用与原创声明（必填项）

1. 合规组件接入声明
   - 文档与图片解析：使用 MinerU 官方在线 API
   - 音视频解析：使用飞书妙记或通义听悟 API
   - 前端框架：React、Vite、TypeScript、自定义 CSS
   - 后端框架：FastAPI、Uvicorn
   - 数据组件：Qdrant、SQLite
2. 核心智能引擎大模型
   - 纯文本向量模型：阿里云百炼 `text-embedding-v4`
   - 文本重排序模型：阿里云百炼 `qwen3-rerank`
   - 视觉小模型：阿里云百炼 `qwen3.5-flash`
   - 最终多模态问答模型：阿里云百炼 `qwen3.5-plus`
3. 自研部分
   - MinerU 返回压缩包解析逻辑
   - 标准化 IR 中间格式
   - DOM 重建与 `header_path`
   - 结构感知切片
   - 提示词编排与溯源展示逻辑

## 5、结论

本项目的推荐路线不是“让所有模态直接共享同一向量空间”，而是：

1. 用专业解析服务先把复杂资料高质量结构化
2. 用自研系统重建文档结构、标准化 IR、完成结构感知切片
3. 用成熟的纯文本嵌入模型做主检索
4. 用重排序提高命中质量
5. 用多模态旗舰模型在最终回答阶段综合理解图片、表格与公式

这条路线更符合当前模型能力现状，也更适合在有限开发时长下完成一个效果可靠、逻辑完整、适合课程展示和简历介绍的大学生 AI 知识库项目。
